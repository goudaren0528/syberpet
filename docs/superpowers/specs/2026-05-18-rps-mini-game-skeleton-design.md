# SyberPet 猜拳 Mini-Game Skeleton Design

## Context

SyberPet 当前已经完成体验骨架中的大部分能力：轻量输入条、宠物气泡回复、thinking → streaming 对话节奏、只读历史面板，以及规则型 pet needs 增长。根据 `.sybermem/requirements/2026-05-17-002-experience-skeleton-chat-history-and-light-play.md`，当前剩余的核心项是补齐一个“聊天式互动玩法样板”，第一版明确采用猜拳。

当前项目的主交互形态是“右键菜单 + 输入条 + 气泡反馈”，因此猜拳也应沿用这一交互链路，不重新引入重聊天面板，也不把玩法判定交给 LLM。

## Goal

补齐一个最小可用的猜拳玩法骨架，使用户可以从右键菜单进入玩法、在输入条中出拳、收到宠物气泡反馈，并获得轻量奖励，同时不破坏现有聊天、设置、历史回看与 pet needs 持久化行为。

## Non-Goals

本次明确不做：

- 多回合玩法
- 连胜/连败机制
- 排行榜或积分系统
- 专门的猜拳面板或复杂 UI
- LLM 参与出拳或结果判断
- 模糊到无限宽松的自然语言理解

## Design

### 1. Entry and UX

在 `renderer/App.tsx` 的现有右键菜单中新增一个入口：`互动玩法（猜拳）`。

用户点击后：

1. 关闭右键菜单
2. 进入“猜拳等待出拳”状态
3. 通过现有宠物气泡提示：`来玩猜拳吧，请输入石头 / 剪刀 / 布`
4. 将 `InputBar` 的 placeholder 明确切换为：`请输入石头 / 剪刀 / 布`

这样可以让玩法入口和现有轻交互风格保持一致，并通过占位提示降低用户忘记当前处于玩法态的概率。

### 2. State Shape

在 `renderer/store/state.ts` 中增加一个轻量玩法状态字段，用于表达“当前是否处于猜拳等待态”。第一版只需要能表达：

- 不在猜拳中
- 正在等待用户出拳

不需要记录多轮历史、连胜信息或复杂结果对象。猜拳结果本身通过现有 `messages` 和 `petBubble` 表达即可，不把多余数据堆进 store。

Store 还需要配套最小的状态切换动作，例如：

- 开始猜拳
- 取消/结束猜拳

### 3. Input Handling

`renderer/ui/InputBar.tsx` 中现有的 `sendMessage()` 是主发送路径，因此猜拳输入也在这里分流，而不是新开一条发送通道。

处理原则：

- 如果当前不在猜拳状态，保持现有聊天逻辑不变：
  - 写入用户消息
  - 应用互动奖励
  - 进入 streaming/thinking 链路
  - 调用 `sendToAgent`
- 如果当前处于猜拳等待态，则改走本地玩法逻辑：
  - 写入用户消息
  - 不调用 `sendToAgent`
  - 解析输入是否为合法拳型
  - 合法则立即结算一局并退出猜拳状态
  - 非法则用气泡提示重新输入，并保持猜拳状态

这样可以复用现有输入条与消息记录能力，同时避免玩法和聊天链路互相污染。

### 4. Rule Layer

新增一个纯逻辑文件，负责猜拳相关规则，避免把归一化、胜负判断、奖励计算直接塞进 UI。

建议职责包括：

- 将用户输入归一化为标准拳型：`rock | paper | scissors`
- 兼容约定范围内的常见别名
- 随机生成宠物拳型
- 判断结果：赢 / 输 / 平
- 生成结果文本
- 返回本轮奖励增量

兼容范围采用已确认的“适度宽松”方案：

- 中文标准输入：`石头 / 剪刀 / 布`
- 中文常见别名：如 `拳头 / 剪子`
- 英文输入：`rock / paper / scissors`

但第一版不做无限宽松匹配，避免把普通聊天文本误判成出拳。

### 5. Reward Model

奖励采用需求文档中已经确定的最小模型：

- 只要完成一局：`mood +3`
- 若用户获胜：额外 `energy +2`
- 平局和失败：不惩罚

奖励最终仍通过现有 `applyInteractionReward()` 进入 pet needs 更新路径，以保持 needs 写入、clamp 与持久化节奏的一致性。

### 6. User Feedback

结果反馈仍然沿用现有宠物气泡，不新增专门的玩法面板。

建议反馈风格：

- 开局提示：`来玩猜拳吧，请输入石头 / 剪刀 / 布`
- 非法输入：`这不是有效出拳哦，请输入石头 / 剪刀 / 布`
- 合法结算：明确展示双方出拳和结果，例如：
  - `我出布，你出石头，这局我赢啦～`
  - `我出剪刀，你赢啦，真厉害！`
  - `都是布，平局～`

反馈原则是“轻、直接、可读”，不做额外 UI 扩展。

## File Impact

### Modify
- `renderer/App.tsx`
  - 在右键菜单中新增猜拳入口，并触发开始猜拳
- `renderer/store/state.ts`
  - 增加轻量猜拳状态与最小状态切换 action
- `renderer/ui/InputBar.tsx`
  - 在主发送路径中分流猜拳输入，并在玩法态切换 placeholder

### Create
- `renderer/pet/rpsGame.ts`
  - 猜拳输入归一化、随机出拳、胜负判断、结果文本与奖励计算

## Verification

功能验证至少覆盖：

1. 右键菜单可见 `互动玩法（猜拳）` 入口
2. 点击入口后会显示开局提示气泡
3. 玩法态下输入框 placeholder 切换为 `请输入石头 / 剪刀 / 布`
4. 合法输入可完整结算一局并退出玩法态
5. 非法输入不会调用 agent，且仍保持玩法态
6. 完成一局后 `mood +3`
7. 用户赢时额外 `energy +2`
8. 普通聊天发送路径不受影响
9. 设置入口、历史面板、气泡显示与 pet needs 持久化不受破坏

建议采用 TDD：

- 先为纯规则层写失败测试（输入归一化、胜负判断、奖励）
- 再补最小实现
- 最后做 UI/状态接线验证

## Recommendation

本次版本应被实现为一个“尽量本地、尽量轻量、尽量不打扰原有聊天链路”的猜拳样板能力。它不是独立子系统，而是对现有桌宠交互骨架的一次最小扩展：证明 SyberPet 不仅能聊天，也能在同一输入通道中承载一轮轻互动玩法。
